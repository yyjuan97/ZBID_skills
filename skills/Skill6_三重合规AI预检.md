## 一、核查维度一：废标缺项核查

### 1.1 核查方法

| 步骤 | 操作 |
|------|------|
| 1 | 逐条读取{项目}/01_解析与基准/废标核查表.md中每条废标条款 |
| 2 | 读取{项目}/02_目录与提示词/风控台账.md中每条【废标风险提示】 |
| 3 | 在{项目}/04_合并与质检/合并初稿.md中全文检索，判断该废标条款对应的内容是否在正文中出现 |
| 4 | 记录核查结果 |

### 1.2 缺项判定标准

| 情境 | 判定 | 风险等级 |
|------|------|---------|
| 废标条款要求提供某资质，正文中完全未提及该资质 | 缺项 | 高危 |
| 废标条款要求某项承诺，正文中有相关描述但未明确做出承诺 | 部分缺项 | 高危 |
| 废标条款要求某格式内容（如"加盖公章"），正文无对应说明 | 格式缺项 | 中风险 |
| 正文有对应内容，但使用了占位符【【人工补充：XXX】】未填补 | 占位未填补 | 高危 |
| 正文有对应内容且已完整描述 | 通过 | — |

### 1.3 检索判定技巧

- 不仅检索关键词，还需检索同义表述（如"资质"还需检索"证书""认证""许可"）；
- 不可仅凭标题判断，需深入段落内容检查；
- ★号条款/实质性要求必须100%覆盖，不设容错。

## 二、核查维度二：评分漏项核查

### 2.1 核查方法

| 步骤 | 操作 |
|------|------|
| 1 | 逐条读取{项目}/01_解析与基准/评分对标表.md中每条评分条目 |
| 2 | 读取{项目}/02_目录与提示词/风控台账.md中每条【对应评分条目】 |
| 3 | 在{项目}/04_合并与质检/合并初稿.md中定位该评分条目对应的章节 |
| 4 | 判断正文中是否对该评分条目做出了明确应答 |

### 2.2 漏项判定标准

| 情境 | 判定 | 风险等级 |
|------|------|---------|
| 评分条目要求描述XX方案，正文对应章节完全未提及 | 漏项 | 中风险 |
| 评分条目要求描述XX方案，正文有提及但未展开论述 | 弱应答 | 中风险 |
| 评分条目要求提供佐证材料，正文仅有描述无佐证材料说明 | 佐证缺失 | 中风险 |
| 评分条目要求满足XX条件，正文有对应应答且论述完整 | 通过 | — |
| 评分条目分档评分（如"满足A得3分，满足B得5分"），正文仅应答了低档条件 | 未争取高分 | 低风险 |

### 2.3 应答完整性判定

- "明确应答"标准：正文中有专门段落/小节回应该评分条目的要求；
- "弱应答"标准：正文中有提及，但仅一两句带过，未按评分条件逐条展开；
- 核查时需标注漏项在正文中的具体位置（章节编号+段落序号）。

## 三、核查维度三：技术偏离核查

### 3.1 核查方法

| 步骤 | 操作 |
|------|------|
| 1 | 逐条读取{项目}/01_解析与基准/技术响应表.md中每条技术需求 |
| 2 | 读取{项目}/02_目录与提示词/风控台账.md中每条【招标技术硬性约束】 |
| 3 | 在{项目}/04_合并与质检/合并初稿.md中定位该技术需求对应的正文段落 |
| 4 | 比对正文描述与招标参数要求是否一致 |

### 3.2 偏离判定标准

| 情境 | 判定 | 风险等级 |
|------|------|---------|
| 正文描述的参数明显低于招标要求（如招标≥500TB，正文写300TB） | 显性负偏离 | 高危 |
| 正文描述与招标要求措辞一致但增加了限定条件 | 隐性负偏离 | 高危 |
| 正文使用了模糊表述回避具体参数（如"大容量存储"替代"≥500TB"） | 参数模糊 | 中风险 |
| 正文描述与招标要求一致 | 通过 | — |
| 正文描述优于招标要求 | 正偏离 | 低风险（提示即可，非问题） |

### 3.3 隐性负偏离识别方法

- 检查正文是否对招标硬性参数添加了前提条件（如招标要求"7×24小时运维"，正文写"工作日7×24小时运维"）；
- 检查正文是否缩小了招标要求的适用范围（如招标要求"全系统"，正文写"核心系统"）；
- 检查正文是否将招标硬性要求弱化为建议性表述（如招标要求"必须"，正文写"建议""推荐"）。

## 六、整改复检机制

### 6.1 复检触发

用户依据质检报告完成整改后，回复【整改完成】，AI执行复检。

### 6.2 复检范围

| 复检项 | 范围 |
|--------|------|
| 整改项 | 逐一复检用户报告中的每个问题是否已整改 |
| 整改衍生影响 | 检查整改是否引入新问题（如新增段落导致引用失效） |
| 已通过项 | 不复检（除非用户明确要求） |

### 6.3 复检结果

输出文件 `{项目}/04_合并与质检/复检报告.md`，结构同首次质检报告，仅包含：
- 整改项复检结果（通过/未通过+原因）
- 新发现的问题（如有）

### 6.4 复检轮次

- 首次复检后如仍有未通过项，用户继续整改后再次回复【整改完成】；
- 每轮复检均输出新的复检报告；
- 无强制轮次上限，直至全部问题整改通过。

## 七、执行后操作

1. 输出质检报告后，向用户提示："文件 {项目}/04_合并与质检/质检报告.md 已生成，发现高危问题X项、中风险X项、低风险X项，请审阅后进行整改，整改完成后回复【整改完成】"；
2. 用户回复【整改完成】后，读取用户整改后的文件，执行复检；
3. 复检通过后，向用户提示："复检通过，可进入Skill7标准化排版"；
4. 用户确认后进入排版工序。

【禁止行为】
- AI判定方案合格/不合格（仅预警，不判定）

- 跳过任何核查维度

- 在质检报告中使用"合格""不合格"等终审结论性表述
