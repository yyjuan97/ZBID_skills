# Skill2｜生成三份基准文档

【触发口令】调用技能2：生成三份基准文档

【执行前置】Skill1解析结果用户已确认通过

【执行前必须读取】{项目}/01_解析与基准/招标文件解析.md，以及本文件（投标编制规范.md中的角色定位、流转规则、权限红线）

【本步骤产出说明】
本步骤产出三份基准文档，是后续所有工序的数据基础：
1. **评分对标表**：将招标文件中所有技术评分条目逐条列出，标注分值，供步骤3目录排序和步骤7质检使用；
2. **技术响应表**：将招标文件中所有技术要求逐条列出，提取参数要求，是步骤3目录子章节的来源、步骤4提示词逐条绑定的对象、步骤5撰写的对标依据、步骤7技术偏离核查的基准；
3. **废标核查表**：将招标文件中所有技术相关废标条款逐条列出，供步骤7废标缺项核查使用。

【执行行为指引】
1. 向用户说明当前步骤："正在执行步骤2：生成三份基准文档，将基于解析结果生成评分对标表、技术响应表、废标核查表"；
2. 按本Skill规则执行，产出三个文件至项目文件夹；
3. 执行完毕后向用户输出以下信息：
   - 文件已生成：`{项目}/01_解析与基准/评分对标表.md`、`技术响应表.md`、`废标核查表.md`
   - 概要：评分对标表X条、技术响应表X条（其中★号X条）、废标核查表X条
   - 技术响应表校验结果：覆盖技术需求规范区条目X/X项，★号条目X/X项，缺失项已补充
   - 提示用户："请审阅文件后回复【确认通过】进入步骤3，或【驳回重做：XX问题】重新生成"
4. 收到【确认通过】后，提醒用户下一步操作："请输入【调用技能3：标书分级目录搭建】"
5. 收到【驳回重做】后，依据驳回说明重新生成，不得沿用旧版内容。

## 一、通用填充规则

### 1.1 原文摘录规则

- 逐字保留招标原文关键词句，不主观概括；
- 原文过长时，可截取核心条款句，省略上下文铺垫语，但不得丢失关键限定词（如"以上""以下""不少于""不超过""不少于""至少"）；
- 截取后用省略号标注，如"……不少于5年运维经验……"。

### 1.2 AI填充范围硬性红线

- AI只允许填充规定列，绝不可越列填写；
- AI不可对空白列填写"待补充""详见"等任何内容，留空即为留空；
- AI不可在表格底部添加汇总行、说明行。

## 二、表格1：评分项对标清单

### 2.1 固定列结构

| 序号 | 招标原文评分条款 | 分值 | 投标响应方向 | 佐证资料 | 人工备注 |
|------|-----------------|------|-------------|---------|---------|

### 2.2 条目拆分粒度

| 情境 | 拆分规则 |
|------|---------|
| 一条原文包含多个独立评分点 | 拆分为多行，每行一个评分点 |
| 一条原文为同一参数的不同档次（如"满足A得3分，满足B得5分"） | 拆分为多行，每行一个档次 |
| 一条原文仅一个评分点 | 一行对应一条 |

### 2.3 AI填充规则（仅前3列）

**序号列**：
- 格式：P-001、P-002……（P代表评分）；
- 按{项目}/01_解析与基准/招标文件解析.md中"技术评分细则区"的条目编号顺序排列。

**招标原文评分条款列**：
- 从{项目}/01_解析与基准/招标文件解析.md"技术评分细则区"逐条摘录；
- 保留原文核心语句，不主观概括；
- 原文过长时按1.1规则截取。

**分值列**：
- 仅填写数字+单位，如"5分""10分"；
- 分档评分的条目，填写该档分值，如"3分（最高5分）"；
- 原文未明确分值的，填写"原文未明确"。

### 2.4 人工列留白说明

| 列名 | 留白原因 | 人工填写指引 |
|------|---------|-------------|
| 投标响应方向 | 需要结合企业实际情况判断 | 填写本企业计划如何响应该评分项 |
| 佐证资料 | 需要结合企业实际资料库存 | 填写能够提供的证明材料清单 |
| 人工备注 | 人工补充说明 | 填写补充说明、待确认事项等 |

## 三、表格2：技术要求响应清单（⚠️重点易出错表格）

### 3.1 固定列结构

| 序号 | 招标技术原文 | 招标参数要求 | 我方响应选型 | 正/负偏离 | 备注 |
|------|-------------|-------------|-------------|----------|------|

### 3.2 条目拆分粒度（⚠️核心规则，直接决定条目数量是否完整）

**基本原则：每个可独立响应的最小技术要求单元占一行。**

| 情境 | 拆分规则 | 示例 |
|------|---------|------|
| 招标文件用**表格**列出设备参数 | 表格每行拆为一行，设备名作为参数上下文 | "核心交换机"表格有5行参数 → 拆为5行，每行前缀"核心交换机：" |
| 一段原文包含**多个独立参数** | 按参数逐个拆行 | "存储容量≥500TB，并发用户≥1000" → 拆为2行 |
| 一个参数有**多个维度** | 合为一行，各维度用分号分隔 | "响应时间≤3秒；可用性≥99.9%" → 合为1行 |
| 参数带**★号**标注 | ★号必须保留在原文列和参数列中 | "★核心交换机背板带宽≥2Tbps" → 原文列和参数列都保留★ |
| 参数有**"或"关系** | 在参数列标注"（A或B）" | "≥500TB或≥1PB" → 参数列写"存储容量≥500TB（或≥1PB）" |
| 参数是**范围值** | 保留范围表述 | "支持100-200个节点" → 参数列写"节点数100-200" |
| 父项包含**子项清单** | 父项拆为独立行，每个子项也各拆一行 | "3.1 网络设备：3.1.1核心交换机……3.1.2防火墙……" → 父项1行+子项各1行 |
| 招标文件用**列表/编号**罗列要求 | 每个编号项拆为一行 | "①支持XX；②支持YY；③支持ZZ" → 拆为3行 |
| 整段描述性要求（无具体参数） | 整段作为一行，参数列写"原文为定性要求" | "提供完善的运维管理体系" → 1行 |

### 3.3 AI填充规则（仅前3列）

**序号列**：
- 格式：T-001、T-002……（T代表技术）；
- 按{项目}/01_解析与基准/招标文件解析.md中"技术需求规范区"的条目编号顺序排列；
- 同一条目拆分为多行时，子编号递增：T-003a、T-003b……

**招标技术原文列**：
- 从{项目}/01_解析与基准/招标文件解析.md"技术需求规范区"逐条摘录；
- 保留原文核心语句，包括★号标注；
- 拆行时保留足够的上下文：每行都需包含该参数属于哪个设备/系统/模块（如"核心交换机：背板带宽≥2Tbps"，不能只写"背板带宽≥2Tbps"丢失归属）；
- **禁止行为**：不得将多行参数合并为一行概述（如把5个网络设备参数合并写"网络设备参数详见招标文件第X页"）。

**招标参数要求列**：
- 从原文列中提炼量化指标，格式：参数名=参数值；
- 多个参数用分号分隔，示例："背板带宽≥2Tbps；转发率≥1000Mpps"；
- ★号参数必须标注★，示例："★背板带宽≥2Tbps"；
- 非量化要求直接摘录原文要求，示例："提供7×24小时技术支持"；
- 原文无量化指标的，填写"原文为定性要求"；
- **与原文列的分工**：原文列=招标怎么说的（保留原话），参数列=提炼出可对比的指标（方便后续逐项对比响应）。

### 3.4 条目完整性校验（⚠️生成后必须执行）

技术响应表最容易漏条目，生成后必须执行以下校验：

| 校验步骤 | 校验方法 | 不通过处理 |
|---------|---------|-----------|
| 1.逐条回溯 | 将{项目}/01_解析与基准/招标文件解析.md"技术需求规范区"的每条条目编号，逐一核对是否在表格中有对应行 | 缺失条目必须补充为独立行 |
| 2.★号全量检查 | 统计"技术需求规范区"中标注★号的条目总数，核对表格中★号行数是否一致 | ★号条目必须100%覆盖 |
| 3.设备/系统清单核对 | 招标文件中列出了设备名称清单的，核对表格中是否每个设备都有对应行 | 缺失设备必须补充 |
| 4.表格型参数行数核对 | 招标文件中用表格列出参数的，核对表格行数与产出行数是否对应 | 行数不足的必须补充 |

校验结果向用户提示："经校验，技术响应表覆盖技术需求规范区条目X/X项，★号条目X/X项，设备/系统X/X项，缺失项已补充。"

### 3.5 人工列留白说明

| 列名 | 留白原因 | 人工填写指引 |
|------|---------|-------------|
| 我方响应选型 | 需要结合企业产品/方案 | 填写拟采用的产品型号、方案名称、对应参数 |
| 正/负偏离 | AI无权判定偏离 | 根据响应选型对比招标参数，标注正偏离/负偏离/无偏离 |
| 备注 | 人工补充说明 | 填写补充说明、偏离说明等 |

【禁止行为】
- AI自主填写偏离、佐证、满足状态、备注内容
